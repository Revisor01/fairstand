**Algorithmus der Hilfsfunktion:**

```typescript
type FifoLot = { ek_cents: number; quantity: number; moved_at: number };
type FifoItem = {
  id: string;
  article_number: string;
  name: string;
  current_stock: number;
  current_ek_cents: number;
  sold_qty: number;
  withdrawn_qty: number;
  revenue_cents: number;
  cost_cents: number;
  withdrawal_cost_cents: number;
  stock_value_cents: number;    // NEU: FIFO-basierter Bestandswert
  remaining_lots: FifoLot[];   // NEU: verbleibende Chargen
  price_periods: PricePeriod[];
};
type FifoInventoryResult = {
  items: FifoItem[];
  total_stock_value_cents: number;
  total_purchased_cents: number;
};

async function computeFifoInventory(shopId: string, year: number): Promise<FifoInventoryResult>
```

**Schritte innerhalb computeFifoInventory:**

1. Alle Produkte für shopId mit aktuellem EK laden (vorhandene Query 1 aus dem Handler, Zeilen ~175-209).

2. Alle stock_movements für shopId laden, sortiert nach movedAt ASC:
```sql
SELECT product_id, type, quantity, purchase_price_cents, moved_at
FROM stock_movements
WHERE shop_id = {shopId}
ORDER BY product_id, moved_at ASC
```

3. Pro Produkt FIFO-Chargen berechnen (in TypeScript, nicht SQL):
   - Lauf durch alle Movements nach movedAt ASC
   - Typ 'restock' UND `purchase_price_cents IS NOT NULL`: Charge als `{ ek_cents: purchasePriceCents, quantity, moved_at }` in eine Lot-Queue schieben
   - Typ 'adjustment' mit `quantity > 0` UND `purchase_price_cents IS NOT NULL`: ebenfalls als Charge einfügen (Phase 37 ANPASS-02)
   - Typ 'sale' oder 'adjustment' mit `quantity < 0`: Menge aus ältesten Chargen FIFO verbrauchen (|quantity| Stück)
   - Typ 'return': Menge zur neuesten Charge addieren oder neue Charge mit current_ek_cents anlegen
   - Nach Durchlauf: `remaining_lots` = verbleibende Chargen mit quantity > 0

4. Fallback für Produkte ohne Chargen (keine stock_movements mit purchase_price_cents):
   - `remaining_lots = [{ ek_cents: current_ek_cents, quantity: current_stock, moved_at: 0 }]`
   - `stock_value_cents = current_ek_cents * current_stock`

5. FIFO-Bestandswert pro Produkt: `stock_value_cents = sum(lot.ek_cents * lot.quantity for lot in remaining_lots)`

6. Preisperioden-Logik (sold_qty, revenue_cents, cost_cents, withdrawal_cost_cents) bleibt unverändert aus dem bestehenden Handler (Zeilen 248-424). Diese wird in computeFifoInventory integriert.

7. total_purchased_cents: alle restocks + adjustments mit purchase_price_cents > 0 im Jahr summieren:
```sql
SELECT product_id, quantity, purchase_price_cents, moved_at
FROM stock_movements
WHERE shop_id = {shopId}
  AND moved_at >= {yearStart}
  AND moved_at < {yearEnd}
  AND purchase_price_cents IS NOT NULL
  AND quantity > 0
  AND type IN ('restock', 'adjustment')
```

**Im /reports/inventory Handler (Zeile ~164-472):**
- Handler-Body durch `const result = await computeFifoInventory(shopId, y);` ersetzen
- Response bleibt strukturell gleich, aber `total_stock_value_cents` kommt jetzt aus FIFO (`result.total_stock_value_cents`), und jedes Item bekommt zusätzlich `stock_value_cents` und `remaining_lots`

**WICHTIG:** Bestehende Felder `price_periods`, `sold_qty`, `withdrawn_qty`, `revenue_cents`, `cost_cents`, `withdrawal_cost_cents` bleiben erhalten — die Preisperioden-Logik wird in computeFifoInventory integriert, nicht entfernt.
